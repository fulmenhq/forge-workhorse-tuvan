# Release Checklist

Standard checklist for forge-workhorse-tuvan releases to ensure consistency and quality.

## Pre-Release Phase

### Version Planning

- [ ] Feature briefs completed in `.plans/active/<version>/`
- [ ] All planned features implemented and tested
- [ ] Breaking changes documented
- [ ] Migration guide written (if applicable)
- [ ] Version number decided (semantic versioning: MAJOR.MINOR.PATCH)

### Code Quality

- [ ] All tests passing: `make test`
- [ ] Code formatted: `make fmt` + `goneat format --types yaml,markdown`
- [ ] Lint checks clean: `make lint`
- [ ] Type checking passes: `make typecheck`
- [ ] Application builds: `make build`
- [ ] Manual smoke tests completed:
  - [ ] `./bin/tuvan version`
  - [ ] `./bin/tuvan serve` (starts without errors)
  - [ ] `./bin/tuvan health`
  - [ ] Graceful shutdown (Ctrl+C)

### Documentation

- [ ] `CHANGELOG.md` updated with new version section
- [ ] `RELEASE_NOTES.md` updated with new version section
- [ ] `docs/releases/v<version>.md` created
- [ ] `README.md` reviewed and updated
- [ ] CDRL guide reflects any workflow changes
- [ ] Feature documentation added to `docs/` (if applicable)
- [ ] `.fulmen/app.yaml` comments accurate for template users

### Dependencies

- [ ] Dependencies reviewed: `bun outdated`
- [ ] No security vulnerabilities: `goneat assess --categories security,dependencies`
- [ ] License compliance verified: `npx license-checker --summary`
- [ ] `bun.lock` up to date
- [ ] Zero copyleft licenses in production dependencies

## Release Preparation

### Version Updates

> [!IMPORTANT]
> **Never hand-edit `VERSION` or `package.json`'s `version`.** Always bump
> through the make targets so the version propagates to every source of truth
> (currently `VERSION` → `package.json` via `scripts/sync-version.ts`, and the
> embedded identity in `dist/` at build time). Hand-editing `VERSION` alone
> leaves `package.json` stale — which then mislabels the `npm pack` tarball in
> the release workflow. The `release-guard-tag-version` check below exists to
> catch exactly that drift.

- [ ] Bump the version via a make target (propagates to `package.json`):
  - `make version-set VERSION=x.y.z`, or
  - `make version-bump-patch` (or `-minor` / `-major`)
- [ ] Confirm `VERSION` and `package.json` agree: `make release-guard-tag-version`
- [ ] Search for hardcoded version references

### Git Hygiene

- [ ] All changes committed
- [ ] Commit messages follow attribution standard
- [ ] No uncommitted changes: `git status` clean
- [ ] All commits have proper trailers:
  - [ ] `Generated by <agent> under supervision of @3leapsdave`
  - [ ] `Co-Authored-By: <agent> <email>`
  - [ ] `Committer-of-Record: <human> <email> [@handle]`

### Final Validation

- [ ] Full quality gate: `make check-all` (lint + typecheck + test)
- [ ] Release readiness (includes the version guard): `make release-check`
- [ ] Build succeeds: `make build`
- [ ] Fresh clone test: Clone repo fresh, run `make bootstrap && make check-all`
- [ ] CDRL workflow test: Simulate user refitting template
  - [ ] Clone template
  - [ ] Update `.fulmen/app.yaml`
  - [ ] Update `package.json` name and metadata
  - [ ] Run `make bootstrap && make build` - verify new identity used

## Release Execution

### Tagging

- [ ] Push commits: `git push origin main`
- [ ] Final guard before tagging — VERSION/package.json/tag agree:
      `TUVAN_RELEASE_TAG=v<version> make release-guard-tag-version`
- [ ] Create annotated git tag: `git tag -a v<version> -m "Release v<version>"`
- [ ] Push tag: `git push origin v<version>` (the release workflow re-runs the same guard)
- [ ] Wait for CI release workflow to complete (creates **draft** release with unsigned artifacts)

### Release Artifacts & Signing

Follow the Fulmen "manifest-only" provenance pattern:

- CI builds cross-platform binaries and npm tarball
- CI generates SHA256 + SHA512 manifests
- CI creates a draft GitHub release with all artifacts
- Maintainer downloads, verifies, signs, and uploads provenance

**Signing workflow (run locally after CI creates draft):**

- [ ] Download CI-built artifacts:

  ```bash
  make release-clean
  make release-download TAG=v<version>
  ```

- [ ] Verify checksums match CI-generated manifests:

  ```bash
  make release-verify-checksums
  ```

- [ ] Sign manifests (minisign required; PGP optional):

  ```bash
  export TUVAN_MINISIGN_KEY=/path/to/tuvan.key
  export TUVAN_MINISIGN_PUB=/path/to/tuvan.pub
  export TUVAN_PGP_KEY_ID="security@fulmenhq.dev"     # optional
  export TUVAN_GPG_HOMEDIR=/path/to/gnupg-fulmenhq     # required if PGP

  # Ensure GPG can prompt for passphrase in this terminal
  export GPG_TTY="$(tty)"
  gpg-connect-agent updatestartuptty /bye

  make release-sign TAG=v<version>
  ```

- [ ] Export public keys: `make release-export-keys`
- [ ] Upload provenance and publish release:

  ```bash
  make release-upload-provenance TAG=v<version>
  ```

**CDRL note**: Downstream users should refit the env var prefix (e.g., `TUVAN_MINISIGN_KEY` -> `MYAPI_MINISIGN_KEY`). See the `SIGNING_ENV_PREFIX` variable in the Makefile.

### Verification

- [ ] GitHub release page shows all expected artifacts:
  - [ ] Cross-platform binaries (5 targets)
  - [ ] npm tarball (`.tgz`)
  - [ ] `SHA256SUMS`, `SHA512SUMS`
  - [ ] Signature files (`.minisig`, optionally `.asc`)
  - [ ] Public key files
- [ ] Download and verify at least one binary works
- [ ] Verify checksums: `sha256sum -c SHA256SUMS`

## Post-Release

### Communication

- [ ] Announce release in Mattermost `#agents-tuvan`
- [ ] Update project README badges (if applicable)
- [ ] Notify tsfulmen team if integration patterns changed

### Housekeeping

- [ ] Archive old release notes (keep only latest 3 in RELEASE_NOTES.md)
- [ ] Clean up `.plans/active/<old-version>/` (move to `.plans/archive/` if needed)
- [ ] Update template usage examples (if applicable)
- [ ] Plan next version features
- [ ] Clean release artifacts: `make release-clean`

### Monitoring

- [ ] Monitor GitHub issues for release-related bugs
- [ ] Check template adoption metrics (if available)
- [ ] Gather feedback from CDRL users

## Version-Specific Checklists

### For Major Releases (x.0.0)

- [ ] Breaking changes documented with upgrade guide
- [ ] Deprecation warnings added to old APIs
- [ ] Migration scripts provided (if complex changes)
- [ ] Announcement prepared for ecosystem impact
- [ ] Backward compatibility strategy documented

### For Minor Releases (0.x.0)

- [ ] New features documented with examples
- [ ] CDRL workflow validated with new features
- [ ] Integration tests cover new functionality
- [ ] Feature flags documented (if applicable)

### For Patch Releases (0.0.x)

- [ ] Bug fixes documented with issue references
- [ ] Regression tests added for fixed bugs
- [ ] Security patches highlighted (if applicable)
- [ ] No new features or breaking changes

## Emergency Hotfix Process

### Hotfix Identification

- [ ] Critical bug or security issue identified
- [ ] Severity assessed (production-impacting?)
- [ ] Hotfix branch created: `hotfix/v<version>`

### Rapid Development

- [ ] Minimal fix implemented
- [ ] Tests added for regression prevention
- [ ] Code review expedited (but not skipped)
- [ ] Quality gates still enforced (no shortcuts)

### Hotfix Release

- [ ] Version bumped (patch level)
- [ ] Release notes include "Hotfix" designation
- [ ] Tag pushed immediately after merge
- [ ] Users notified of critical update

### Post-Hotfix

- [ ] Root cause analysis documented
- [ ] Process improvements identified
- [ ] Monitoring enhanced to catch similar issues

## npm Publishing (Optional)

If you choose to publish to npm (not used by tuvan template itself):

- [ ] Verify `publish-npm` GitHub Environment exists with deployment protection
- [ ] Verify npm trusted publishing is configured for your package
- [ ] Run publish workflow: Actions -> "Publish to npm" -> Run workflow
- [ ] Verify package available: `npm view @your-scope/your-package@<version>`
- [ ] Test installation: `npm install @your-scope/your-package@<version>`

See `.github/workflows/publish.yml` for the OIDC publishing workflow.

## Notes

- VERSION file is the single source of truth — always use `make version-bump-*`
- This checklist may evolve with project maturity
- Some items may not apply to all releases (use judgment)
- Prioritize quality over speed — never skip tests or code review
- When in doubt, consult @3leapsdave before proceeding
