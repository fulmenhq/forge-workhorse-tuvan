## Session Startup Protocol

1. **Context Review**
   - **REQUIRED**: Read `Makefile` to understand available targets and build workflow
   - Read `MAINTAINERS.md`, `REPOSITORY_SAFETY_PROTOCOLS.md` (when created), `README.md`, and `.fulmen/app.yaml` (when created)
   - Review Crucible standards in `docs/` directory (via tsfulmen)
   - **CRITICAL**: Understand that this is a template - consumers will CDRL (Clone → Degit → Refit → Launch) this repo

2. **Environment Check**
   - Confirm required tools (`node >= 20.0.0`, `bun >= 1.0.0`, `make`) are available
   - Run `make bootstrap` if tools are missing

3. **Plan**
   - Outline steps in `.plans/` (gitignored) or within the session transcript before modifying files
   - For major features, create feature briefs in `.plans/active/<version>/`

4. **Quality Assurance**
   - Run `make test` before every commit
   - Verify code formatting: `make fmt`
   - Run linting: `make lint` or `make check-all`
   - Ensure application builds: `make build`
   - Run type checking: `make typecheck`
   - Test CLI commands manually when adding/modifying

5. **Execute Safely**
   - Use prescribed Make targets: `make test`, `make fmt`, `make lint`, `make build`, `make typecheck`
   - Follow TypeScript best practices and idioms (strict mode)
   - Maintain template flexibility for CDRL consumers

6. **Attribution**
   - Follow [Agentic Attribution Standard](https://github.com/fulmenhq/crucible/blob/main/docs/standards/agentic-attribution.md) for commit messages

7. **Supervision**
   - Confirm human reviewer/supervisor availability before merging or publishing

## Operational Guidelines

### DO

- **Quality First**: Always run `make test`, `make typecheck`, and `make lint` before commits
- **Use Make Targets**: Prefer `make test`, `make build`, `make run` over raw commands
- **Follow Workhorse Standard**: Adhere to Fulmen Forge Workhorse Standard (see Crucible docs)
- **Maintain Test Coverage**: Add tests for all new functionality (target: 80%+)
- **Document for CDRL Users**: Update README and docs for template consumers
- **Use App Identity**: Never hardcode app name, env var prefix, or config paths
- **Integrate tsfulmen**: Use tsfulmen modules for logging, config, telemetry, identity
- **Production Patterns**: Graceful shutdown, config reload, observability, error handling
- **Type Safety**: Use TypeScript strict mode, avoid `any` types

### DO NOT

- **Edit App Identity Directly**: `.fulmen/app.yaml` is the template identity - CDRL users will customize
- **Commit Planning Files**: `.plans/` is permanently gitignored—never add, stage, or attempt to commit anything under this directory
- **Skip Tests**: Never commit code without passing tests
- **Ignore Linting**: All code must pass linting before commit
- **Commit Without Formatting**: Run `make fmt` before every commit
- **Skip Type Checking**: TypeScript must compile with zero errors in strict mode
- **Hardcode Application Names**: Use app identity APIs for all app-specific values
- **Break CDRL Workflow**: Changes should make refitting easier, not harder
- **Store Secrets**: Never commit API keys, tokens, or credentials
- **Use `any` Types**: Maintain strict TypeScript typing throughout
- **Use `bun test`**: ALWAYS use `bun run test` (Vitest) instead of `bun test` (Bun's test runner) - this project requires Vitest for enterprise-scale testing

## Workhorse-Specific Guidelines

### Template Philosophy

This is a **template repository** that users will CDRL:

- **Clone**: `git clone` to local machine
- **Degit**: Remove git history
- **Refit**: Customize via `.fulmen/app.yaml` and package.json
- **Launch**: Run their application

All changes must consider the CDRL workflow and make it easier for users to customize the template.

### Key Template Features

- **App Identity**: Single-file identity management (`.fulmen/app.yaml`)
- **CLI Framework**: Commander.js-based commands (serve, version, health, envinfo, doctor)
- **HTTP Server**: Fastify router with standard endpoints (/health, /version, /metrics)
- **Observability**: TSFulmen logging (SIMPLE/STRUCTURED/ENTERPRISE profiles) and Prometheus metrics
- **Configuration**: Three-layer config (defaults → file → env vars)
- **Production Ready**: Graceful shutdown, signal handling, error handling
- **Type Safety**: Strict TypeScript with comprehensive type coverage

### Integration Priorities

1. **App Identity** - Eliminate hardcoding
2. **Signal Handling** - Graceful shutdown, config reload, double-tap force quit
3. **Exit Codes** - Standardized foundry exit codes for operations
4. **Observability** - Structured logging, metrics, tracing
5. **Testing** - Comprehensive test coverage with Vitest
6. **Documentation** - Clear CDRL instructions for consumers

## TypeScript-Specific Guidelines

### Code Quality Standards

- **Strict Mode**: All TypeScript code must compile with `strict: true`
- **No Implicit Any**: Explicitly type all function parameters and returns
- **Null Safety**: Use strict null checks, prefer explicit undefined over null
- **Readonly**: Use `readonly` for immutable data structures
- **Type Inference**: Leverage TypeScript's inference, but be explicit when clarity improves

### Testing Standards

- **Vitest**: Use Vitest for all testing
- **Coverage**: Maintain 80%+ test coverage
- **Type Tests**: Test type correctness, not just runtime behavior
- **Integration Tests**: Test HTTP endpoints end-to-end
- **Mock Strategy**: Use Vitest's native mocking capabilities

### Performance Considerations

- **Async/Await**: Use async/await for all asynchronous operations
- **Stream Processing**: Leverage Node.js streams for large data
- **Event Loop**: Avoid blocking the event loop with heavy computation
- **Memory**: Monitor memory usage, use WeakMap/WeakSet for caches

### Security Best Practices

- **Input Validation**: Validate all external inputs with schemas
- **SQL Injection**: Use parameterized queries (if using databases)
- **XSS Prevention**: Sanitize all user-generated content
- **CORS**: Configure CORS appropriately for your use case
- **Helmet**: Use @fastify/helmet for security headers
